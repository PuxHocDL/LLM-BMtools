Setup Guide

A step-by-step walkthrough — written for someone who has never run an LLM benchmark before — for installing this project, configuring an LLM, and reproducing both the paper baseline and our three enhancements.

In a hurry?

If you already know what you're doing, jump to §6 (baseline) or §7 (enhancements). Everything before that is installation.

What you'll have at the end

After following this guide you'll have:

  • A working Python virtual environment with all dependencies installed.
  • A configured LLM endpoint (cloud or local Ollama).
  • Baseline predictions for at least one model on at least one setup, scored with EM, Contains, and (optionally) LLM-as-a-judge.
  • Enhanced predictions using our HeuristicPlus pruning, Plan-and-Solve, and Self-Correction methods.
  • A summary table you can compare against the numbers in §Result tables.

1. Clone the repository

1
Get the code onto your machine

Open a terminal (PowerShell on Windows, Terminal on macOS/Linux) and:

git clone <repo-url> CS222_final
cd CS222_final

If your professor handed you a ZIP instead, unzip it and cd into the unzipped folder.

2. Create a Python virtual environment

2
Isolate this project's dependencies

A virtual environment keeps the packages used here separate from your global Python install. Pick your OS:

python -m venv .venv
.venv\Scripts\Activate.ps1

If PowerShell complains about "running scripts is disabled", run once: Set-ExecutionPolicy -Scope CurrentUser RemoteSigned.

python3 -m venv .venv
source .venv/bin/activate

Your prompt should now show (.venv) at the start. That means the venv is active. Every later command in this guide assumes the venv is active.

3. Install dependencies

3
Install the libraries used by the project

There are two requirement files — install both:

pip install --upgrade pip
pip install -r requirements.txt
pip install -r baseline/requirements.txt

The first one covers the enhancement methods (root main.py). The second covers the paper baseline (baseline/run_baseline.py).

If you get ModuleNotFoundError: No module named 'genai'

The baseline scripts conditionally import an IBM-internal genai binding that most users don't need. Two options:

  1. Install it: pip install ibm-generative-ai
  2. Or comment out the import in baseline/generate_qa_pairs/tasks/utils.py at the line starting with from genai.extensions... — we only use the OpenAI / Ollama branches.

4. Pick an LLM provider

You need at least one LLM endpoint that the scripts can talk to. The project supports any provider that exposes an OpenAI-compatible HTTP API. Three common choices:

4a — Any OpenAI-compatible endpoint (recommended)

A
Use vLLM, OpenAI, Modal, Together, Anyscale, etc.

This is what the enhancement methods at the project root use. Copy the example config and fill it in:

cp config.example.yaml config.yaml

Open config.yaml and set api_base and api_key for each of granite, gptoss, devstral, and the judge model llama. If you only have one endpoint, you can point all four entries at the same endpoint and just change the model field.

agents:
  granite:
    model: ibm-granite/granite-3.3-8b-instruct
    api_base: https://YOUR_ENDPOINT/v1
    api_key: YOUR_API_KEY
    temperature: 0.0
    timeout: 180.0
  # ...
Tip

config.yaml is git-ignored. Your keys never get committed.

4b — Local Ollama (no cloud key needed)

B
Run open-weight models on your own machine

Install Ollama, then pull the three models we used:

ollama pull granite3.3:8b
ollama pull gpt-oss:20b
ollama pull devstral:24b

Ollama exposes an OpenAI-compatible endpoint at http://localhost:11434/v1. Point config.yaml at it:

agents:
  granite:
    model: granite3.3:8b
    api_base: http://localhost:11434/v1
    api_key: ollama              # any non-empty string works
    temperature: 0.0
    timeout: 600.0

For the baseline scripts (baseline/), instead set LLM_PROVIDER=local in baseline/.env.

4c — Azure OpenAI (paper's setup)

C
Use the same setup as Kate et al.

Only relevant if you have an Azure OpenAI subscription. In baseline/.env set:

LLM_PROVIDER=azure
AZURE_OPENAI_API_KEY=sk-...
AZURE_ENDPOINT=https://your-resource.openai.azure.com/openai/deployments/{model_id}/chat/completions?api-version={api_version}

5. Get the ToolJSON benchmark data

5
Make sure the QA pairs and API responses are in place

The benchmark already lives under data/toolJSONprocessing/generate_qa_pairs/data/. After cloning, verify the layout:

data/toolJSONprocessing/generate_qa_pairs/data/
  qa_pairs/        # one JSON file of QA pairs per endpoint
  api_responses/   # the raw API JSON responses
  schemas/         # the JSON schemas

If any folder is empty, regenerate it by following the instructions in data/toolJSONprocessing/generate_qa_pairs/README.md (uses ComplexFuncBench as a source).

6. Re-run the paper baseline

The baseline/ folder is a self-contained copy of the original scripts from Kate et al. — extracted so you can prove the baseline reproduces. The run_baseline.py wrapper makes it a single command per (setup, model) pair.

6a
Smoke test (≈30 samples, <5 minutes)
cd baseline
cp example.env .env       # edit this file with your provider creds
python run_baseline.py --setup answer --model ibm-granite/granite-3.3-8b-instruct --limit 5

You should see one predictions.json per endpoint appear under baseline/experimental_scripts/results/predictions/.

6b
Full run — one column at a time

Each invocation reproduces one cell of the baseline table. The full benchmark is 2,067 samples and will take 10–60 minutes per (setup, model) pair depending on your endpoint's throughput.

# Granite — best in the "Answer" column
python run_baseline.py --setup answer       --model ibm-granite/granite-3.3-8b-instruct

# GPT-OSS-20B — best in "Code + Schema"
python run_baseline.py --setup code_schema  --model openai/gpt-oss-20b

# Devstral — best in "Code + Schema"
python run_baseline.py --setup code_schema  --model mistralai/Devstral-Small-2507

6c
Score the predictions
# EM + Contains (fast, no LLM call)
python run_evaluation.py

# Add LLM-as-a-judge — needs a judge model id reachable via the same provider
python run_evaluation.py --judge meta-llama/llama-3-3-70b-instruct

The summary printed at the end is what you compare against the baseline table.

7. Run our three enhancement methods

7
Use main.py from the project root

Go back to the project root (and make sure your venv is still active):

cd ..
python main.py --method baseline        --model granite --judge llama
python main.py --method pruning         --model granite
python main.py --method plan_solve      --model granite
python main.py --method self_correction --model gptoss

The --model argument here is the nickname from config.yaml (granite, gptoss, devstral, llama), not the full HuggingFace id.

For a quick sanity check, append --limit 20 --skip-judge.

8. Read the results

After every run, two files appear under results/:

  • results/<method>_<model>_per_sample.{csv,json} — one row per QA sample with the predicted answer and each metric.
  • results/<method>_<model>_summary.json — overall EM, Contains, F1, Judge.

Open the summary and compare to the tables below. Numbers within ±0.01 of the reported value are normal — they depend on which judge model you pick and minor LLM nondeterminism.

Troubleshooting

ModuleNotFoundError: No module named 'genai'

See the note in step 3 — either install ibm-generative-ai or comment out the import in baseline/generate_qa_pairs/tasks/utils.py.

"maximum context length exceeded" in baseline outputs

Some JSON responses (Booking — Get Room List With Availability especially) exceed 60K characters and don't fit in an 8K-context model. The predictions file will record "pred_answer": "context length exceeded" for those rows. The enhancement methods at root main.py avoid this by pruning the JSON first — that's exactly the gap (I) HeuristicPlus addresses.

The judge model is returning all False

Make sure your judge model returns "True" / "False" at the very start of its reply. Older Llama-3 70B works well; smaller models sometimes hedge. You can swap judge with --judge <model-id>.

Numbers don't match the table exactly

Two reasons. First, LLM-as-a-judge depends on which judge model you use — even with temperature 0 different judges give slightly different scores. Second, code-generation setups execute model-written code; an off-by-one in formatting can flip EM. As long as you're within a few hundredths of the reported EM, you're reproducing correctly.
